# UI/UXデザイン対話スキル

## このスキルを使うタイミング

- UIコンポーネントのデザインを決めるとき
- タスクファイル作成前（spec-design でUIタスクを生成する前）
- デザインの曖昧さを100%排除したいとき

---

## ⚠️ 重要原則

**デザインの曖昧さは100%残さない**

- 「いい感じに」→ 具体的なサイズ・色・配置を決める
- 「適度に」→ 数値で明確化（16px, 44px, など）
- 「目立つように」→ どのくらい目立つ？他の要素との比率は？

**参考:** UX心理学の47原則を質問形式で適用する

## 対話の進め方

### ステップ1: コンポーネントの目的を明確化

```
質問例:
- 「このコンポーネントは何のためにありますか？」
- 「ユーザーはどんな場面で使いますか？」
- 「このコンポーネントで達成したいことは？」

目標: コンポーネントの役割を1文で説明できる状態にする
```

### ステップ2: 使用頻度と重要度を確認

```
質問例:
- 「このコンポーネントは頻繁に使われますか？」
  → 頻繁 = フィッツの法則（大きく、近く）
  → たまに = 通常サイズ

- 「このボタンは重要ですか？」
  → 重要 = プライマリカラー、大きめ
  → 補助的 = セカンダリカラー、通常サイズ
  → 破壊的 = destructive カラー、確認ダイアログ必須

- 「急いで操作しますか？」
  → はい = ドハティの閾値（0.4秒以内のレスポンス）

目標: コンポーネントの優先度を数値化する
```

### ステップ3: 情報の階層を決める

```
質問例:
- 「このコンポーネント内で、一番重要な情報は何ですか？」
- 「次に重要な情報は？」
- 「補足情報は？」

回答を元に視覚的階層を設計:
1. 最重要 → text-lg, font-semibold, text-foreground
2. 副次的 → text-sm, text-muted-foreground
3. 補足 → text-xs, text-muted-foreground/70

目標: 情報の重要度順にリスト化する
```

### ステップ4: UX原則を質問形式で適用

#### ボタン・CTA の場合

```
- 「このボタンは画面のどこに配置しますか？」
  → 右下/中央 = フィッツの法則（親指で届く）
  → 上部 = 通常配置

- 「タップエリアは十分ですか？」
  → モバイル = 最低44x44px（フィッツの法則）

- 「他のボタンと間違えやすくないですか？」
  → 色・サイズで明確に区別
```

#### フォームの場合

```
- 「入力項目は何個ありますか？」
  → 7個以上 = 決断疲れ（ページ分割を検討）
  → 5個以下 = OK

- 「必須項目は何個ですか？」
  → 最小限に（決断疲れを軽減）

- 「デフォルト値は設定しますか？」
  → はい = デフォルト効果を活用

- 「エラーはいつ表示しますか？」
  → 入力中 = リアルタイムフィードバック
  → 送信時 = 一括表示
```

#### カード・リストの場合

```
- 「カードは何枚表示されますか？」
  → 多い = 無限スクロールor ページネーション
  → 少ない = 一覧表示

- 「カード内の情報はどうグループ化しますか？」
  → ゲシュタルト原理（近接・類似）で設計

- 「空の状態はどう見せますか？」
  → アイコン + メッセージ + アクションボタン
```

#### モーダル・ダイアログの場合

```
- 「このモーダルは破壊的操作ですか？」
  → はい = 意図的な壁（確認ダイアログ必須）
  → いいえ = 通常モーダル

- 「Escキーで閉じますか？」
  → はい（推奨）

- 「背景クリックで閉じますか？」
  → 破壊的操作以外は Yes

- 「フォーカスはどこに当てますか？」
  → 最初の入力欄（フォーカス管理）
```

#### ナビゲーションの場合

```
- 「メニュー項目は何個ありますか？」
  → 7個以上 = ヒックの法則（選択肢を減らす）
  → 5-7個 = OK

- 「階層構造は必要ですか？」
  → はい = ドロップダウンで整理

- 「どのくらいの頻度でナビゲーションしますか？」
  → 頻繁 = 固定ナビゲーション
  → たまに = ハンバーガーメニュー
```

### ステップ5: カラーパレットを決める

```
質問例:
- 「このコンポーネントで何を表現したいですか？」
  → 信頼感 = 青系（primary）
  → 活動的 = オレンジ系（secondary）
  → 成功 = 緑系（success）
  → エラー = 赤系（error）

- 「他の要素と区別する必要がありますか？」
  → はい = 異なる色
  → いいえ = 統一色

目標: デザインシステムのカラーパレットから選択
```

### ステップ6: スペーシングを決める

```
質問例:
- 「このコンポーネント内の要素は関連していますか？」
  → 関連 = 近接配置（4px, 8px）
  → 無関連 = 離して配置（16px, 24px）

- 「パディングはどのくらい必要ですか？」
  → ゆったり = p-6（24px）
  → 通常 = p-4（16px）
  → 密 = p-2（8px）

目標: 4pxベースのスケールで数値化
```

### ステップ7: レスポンシブ対応を決める

```
質問例:
- 「モバイルとPCで見た目を変えますか？」
  → はい = ブレークポイント設定（md:768px）

- 「モバイルで一番重要なのは？」
  → タッチターゲット44px以上を確保

- 「横スクロールは必要ですか？」
  → カラムレイアウト = overflow-x-auto
```

## デザイン確定チェックリスト

以下が全て具体的に決まるまで質問を続ける：

### 視覚的階層
- [ ] 最重要情報が明確（サイズ・色・位置）
- [ ] 副次的情報が明確
- [ ] 補足情報が明確

### UX原則適用
- [ ] 該当するUX原則が特定されている
- [ ] 各原則に基づく具体的なデザイン決定がされている

### カラー
- [ ] プライマリカラー決定
- [ ] セカンダリカラー決定（必要な場合）
- [ ] テキストカラー決定

### スペーシング
- [ ] パディング（p-4, p-6, など）
- [ ] マージン（mt-2, mb-4, など）
- [ ] 要素間の間隔（space-y-2, gap-4, など）

### サイズ
- [ ] 幅（min-w-[280px], w-full, など）
- [ ] 高さ（min-h-[44px], h-auto, など）
- [ ] フォントサイズ（text-lg, text-sm, など）

### インタラクション
- [ ] クリック/タップ時の動作
- [ ] ホバー時の見た目
- [ ] フォーカス状態
- [ ] ローディング状態（必要な場合）

### アクセシビリティ
- [ ] ARIA属性（role, aria-label, など）
- [ ] キーボード操作対応
- [ ] タッチターゲットサイズ（44px以上）

### レスポンシブ
- [ ] モバイル（〜768px）での表示
- [ ] タブレット（768px〜1024px）での表示
- [ ] デスクトップ（1024px〜）での表示

## 出力形式

デザインが確定したら、以下の形式で出力する：

```markdown
## {コンポーネント名} デザイン仕様

### 目的
{1文で説明}

### 適用するUX原則
- {原則1}: {具体的な適用方法}
- {原則2}: {具体的な適用方法}

### 視覚的階層
1. 最重要: {情報名}（text-lg, font-semibold, text-foreground）
2. 副次的: {情報名}（text-sm, text-muted-foreground）
3. 補足: {情報名}（text-xs, text-muted-foreground/70）

### カラーパレット
- 背景: bg-{color}
- テキスト: text-{color}
- ボーダー: border-{color}

### スペーシング
- パディング: p-4
- 要素間: space-y-2
- マージン: mb-4

### サイズ
- 幅: min-w-[280px] md:min-w-[320px]
- 高さ: min-h-[44px]
- フォント: text-base md:text-lg

### インタラクション
- クリック: {動作}
- ホバー: {見た目}
- フォーカス: {見た目}

### アクセシビリティ
- ARIA: role="{role}", aria-label="{label}"
- キーボード: Tab, Enter, Escape
- タッチ: 44x44px 以上

### レスポンシブ
- モバイル: {表示方法}
- PC: {表示方法}

---

**デザイン確定OK → ui_task_template.md でタスク生成へ**
```

## よくある失敗

### ❌ 曖昧なまま進む
→ 「いい感じに」「適度に」は全て数値化する

### ❌ ユーザーの回答を鵜呑みにする
→ 「本当にそれで大丈夫ですか？」と確認

### ❌ UX原則を無視する
→ 必ず該当する原則を適用する

### ❌ 一度に全部聞く
→ 1回に1-2個の質問に絞る

---
