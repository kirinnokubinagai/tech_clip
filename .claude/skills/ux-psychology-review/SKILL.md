# UX心理学デザインレビュースキル

https://shokasonjuku.com/ux-psychology の47原則に基づいた包括的なデザインレビューを実施します。

## クイックスタート

デザインレビューを実行：
```bash
# UIコンポーネントのレビュー
Read components/Button.tsx
# または画面全体のレビュー
Glob "app/**/*.tsx"
```

詳細な原則については [UX_PRINCIPLES.md](UX_PRINCIPLES.md) を参照。
チェックリストは [CHECKLIST.md](CHECKLIST.md) を参照。

## Instructions

### 1. 対象の特定
- ReadでUIコンポーネント、ページファイルを確認
- Globでデザイン関連ファイルをパターンマッチ
- スクリーンショットやモックがある場合は視覚的にも確認

### 2. 心理バイアスの評価
以下の観点でレビュー：

#### 確証バイアス
- ✓ ユーザーの期待に反する情報も適切に表示されているか
- ✓ 反対意見や代替案が抑圧されていないか

#### アンカー効果
- ✓ 価格表示や選択肢の順序が適切か
- ✓ 最初に表示される情報が意図的に設計されているか

#### 期待バイアス
- ✓ ユーザーの事前期待と実際の体験に齟齬がないか
- ✓ オンボーディングで適切な期待値設定ができているか

#### 親近性バイアス
- ✓ 一般的なUIパターン（ハンバーガーメニュー、カードレイアウト等）を活用しているか
- ✓ 業界標準から逸脱する場合、明確な理由があるか

### 3. 意思決定への影響評価

#### 決断疲れ
- ✓ 選択肢の数が適切か（7±2の法則）
- ✓ 段階的な意思決定フローになっているか
- ✓ デフォルト値が提供されているか

#### おとり効果
- ✓ プライシングプランで意図的なおとり選択肢があるか
- ✓ ユーザーを誘導しすぎていないか（倫理的配慮）

#### デフォルト効果
- ✓ ユーザーに有益なデフォルト値が設定されているか
- ✓ オプトアウト可能な設計か

#### 損失回避
- ✓ ネガティブなメッセージ（「失う」「なくなる」）が適切に使われているか
- ✓ キャンセル防止で損失フレーミングを活用しているか

### 4. デザイン応用技法の評価

#### ナッジ効果
- ✓ 強制せずにユーザーを望ましい行動に誘導しているか
- ✓ マイクロコピーで適切なナッジがあるか

#### ゲーミフィケーション
- ✓ プログレスバー、バッジ、レベルなどの要素があるか
- ✓ 過度にゲーム化されていないか（目的とのバランス）

#### 段階的開示
- ✓ 情報が必要なタイミングで表示されているか
- ✓ 初期画面が情報過多になっていないか
- ✓ アコーディオン、タブなどで段階的に情報開示しているか

#### 意図的な壁
- ✓ 重要な操作（削除、課金等）に確認ダイアログがあるか
- ✓ クーリングオフ期間が設けられているか

### 5. パフォーマンス・体験の評価

#### ドハティの閾値
- ✓ インタラクションのレスポンスが0.4秒以内か
- ✓ ローディング状態が適切に表示されているか
- ✓ スケルトンスクリーンなどで体感速度を向上させているか

#### ピーク・エンドの法則
- ✓ ユーザー体験の「最高の瞬間」が設計されているか
- ✓ 終了時の体験（サンクスページ、完了メッセージ等）が良好か

#### 労働の錯覚
- ✓ 処理中の進捗が可視化されているか
- ✓ 「検索中...」「分析中...」などのフィードバックがあるか

### 6. 視覚デザインの評価

#### 視覚的階層
- ✓ 情報の重要度が視覚的に明確か
- ✓ タイポグラフィ、色、サイズで階層が表現されているか

#### ゲシュタルト原理
- ✓ 近接、類似、閉合、連続などの原理が活用されているか
- ✓ 関連する要素がグループ化されているか

#### スキューモーフィズム vs フラットデザイン
- ✓ デザイントレンドと機能性のバランスが取れているか
- ✓ アフォーダンス（操作可能性）が明確か

### 7. 社会的証明・信頼性

#### 社会的証明
- ✓ レビュー、評価、利用者数などが表示されているか
- ✓ 第三者の推薦や認証マークがあるか

#### 権威性
- ✓ 専門家の推薦や実績が示されているか
- ✓ 信頼できる情報源が明示されているか

### 8. アクセシビリティ・インクルーシブデザイン

- ✓ 色覚多様性に配慮した配色か
- ✓ キーボード操作が可能か
- ✓ スクリーンリーダー対応（適切なARIA属性）があるか
- ✓ 多様なユーザー層（高齢者、障がい者等）を考慮しているか

## Examples

### Example 1: ECサイトの商品ページレビュー

```bash
# コンポーネントを確認
Read components/ProductDetail.tsx

# レビュー実行
```

**評価ポイント**：
- アンカー効果：元価格と割引価格の表示順序
- 社会的証明：レビュー数、評価の表示
- 損失回避：「残りわずか」「期間限定」表示
- ドハティの閾値：画像読み込み速度
- デフォルト効果：数量、配送オプションのデフォルト値

### Example 2: SaaSダッシュボードレビュー

```bash
# 全画面を確認
Glob "app/dashboard/**/*.tsx"

# レビュー実行
```

**評価ポイント**：
- 段階的開示：情報の表示タイミング
- 視覚的階層：重要指標の強調
- 労働の錯覚：データ処理中の表示
- ゲーミフィケーション：進捗バー、達成度表示
- 親近性バイアス：一般的なダッシュボードパターン準拠

### Example 3: オンボーディングフローレビュー

```bash
# オンボーディング画面を確認
Read app/onboarding/steps/*.tsx

# レビュー実行
```

**評価ポイント**：
- 決断疲れ：ステップ数の適切性
- ピーク・エンドの法則：完了時の体験
- 期待バイアス：事前期待値の設定
- 段階的開示：情報の段階的提示
- ナッジ効果：次のステップへの誘導

## Best Practices

### レビュー前の準備
1. ターゲットユーザーのペルソナを確認
2. ビジネス目標とKPIを理解
3. 競合サービスのUXを調査
4. 既存のユーザーフィードバックを確認

### レビュー時の注意点
- 単に原則に従うだけでなく、ビジネス目標との整合性を確認
- 倫理的配慮：ダークパターンの使用を避ける
- 定量的データ（ヒートマップ、A/Bテスト結果等）との照合
- モバイル/デスクトップ両方の体験を考慮

### レビュー後のフォロー
- 優先順位付けされたアクションアイテムを作成
- 改善効果の測定方法を定義
- デザインシステムへの反映を検討
- 定期的な再レビューをスケジュール

## Notes

- このスキルはデザインの「良し悪し」を判断するのではなく、UX心理学の原則に基づいた改善機会を発見するためのものです
- すべての原則を適用する必要はありません。プロジェクトの性質に応じて適切な原則を選択してください
- ダークパターン（ユーザーを欺く手法）の使用は避け、倫理的なデザインを推奨します
