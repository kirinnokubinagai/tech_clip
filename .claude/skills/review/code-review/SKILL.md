# コードレビュースキル

## レビュー観点

### 1. 正確性
- [ ] ロジックは正しいか
- [ ] エッジケースは考慮されているか
- [ ] エラーハンドリングは適切か

### 2. コーディング規約
- [ ] 命名規則に従っているか
- [ ] 禁止パターン（any, else, console.log）を使っていないか
- [ ] 早期リターンを使っているか
- [ ] JSDocコメントがあるか

### 3. セキュリティ
- [ ] ハードコードされた機密情報がないか
- [ ] SQLインジェクション対策（ORM使用）
- [ ] XSS対策（エスケープ処理）
- [ ] 認証・認可チェックがあるか

### 4. パフォーマンス
- [ ] N+1クエリがないか
- [ ] 不要な再計算がないか
- [ ] 適切なインデックスがあるか

### 5. 保守性
- [ ] 関数は1つのことだけしているか
- [ ] 過度な抽象化がないか
- [ ] テストが書きやすい構造か

### 6. テスト
- [ ] テストカバレッジは80%以上か
- [ ] 正常系・異常系・境界値がテストされているか
- [ ] テスト名は日本語で意図が明確か

## 出力フォーマット

```markdown
## コードレビュー結果

### サマリー
- 全体評価: ⭐⭐⭐⭐☆ (4/5)
- 重大な問題: X件
- 軽微な問題: Y件

### 重大な問題 🔴
| ファイル | 行 | 問題 | 修正案 |
|----------|-----|------|--------|
| ... | ... | ... | ... |

### 軽微な問題 🟡
| ファイル | 行 | 問題 | 修正案 |
|----------|-----|------|--------|
| ... | ... | ... | ... |

### 良い点 ✅
- ...

### 改善提案 💡
- ...
```

改善提案（💡）が1件でもある場合、PR 全体の最終判定で `✅ Approve` や `Approve 相当` を出してはいけない。

## 重大度定義

| レベル | 定義 | 例 |
|--------|------|-----|
| 🔴 重大 | マージ前に修正必須 | セキュリティ脆弱性、バグ |
| 🟡 軽微 | 修正推奨 | 命名、コードスタイル |
| 💡 提案 | 検討してほしい | リファクタ案 |
