# PRレビュースキル

## レビュープロセス

### 1. PR概要確認
- [ ] タイトルがConventional Commitsに従っているか
- [ ] 説明にIssue番号（Closes #XXX）があるか
- [ ] 変更の目的が明確か

### 2. 変更範囲確認
- [ ] 変更ファイル数は適切か（大きすぎないか）
- [ ] 関係ない変更が混入していないか
- [ ] 1つのPRで1つの関心事か

### 3. コード品質
- [ ] コーディング規約に従っているか
- [ ] セキュリティ問題がないか
- [ ] パフォーマンス問題がないか

### 4. テスト
- [ ] テストが追加/更新されているか
- [ ] カバレッジ80%以上か
- [ ] CI/CDが通っているか

### 5. ドキュメント
- [ ] 必要に応じてREADME更新があるか
- [ ] 破壊的変更の場合、マイグレーションガイドがあるか

### 6. マージ可否判定
- [ ] すべてのチェック項目をクリアしているか

## 出力フォーマット

```markdown
## PRレビュー結果

### 基本情報
- PR: #XXX
- 作成者: @username
- ベースブランチ: main
- 変更ファイル数: X

### チェックリスト
| 項目 | 状態 | コメント |
|------|------|----------|
| タイトル | ✅/❌ | ... |
| Issue紐付け | ✅/❌ | ... |
| テスト | ✅/❌ | ... |
| CI/CD | ✅/❌ | ... |

### 判定
- **✅ Approve**: マージ可能
- **🔄 Request Changes**: 修正が必要
- **💬 Comment**: コメントのみ

### コメント
...
```

## マージ基準

### ✅ Approveの条件
- 重大な問題がない
- 軽微な問題がない
- 改善提案がない
- テストが通っている
- コーディング規約を遵守

### 🔄 Request Changesの条件
- セキュリティ脆弱性がある
- テストが不足している
- 重大なバグがある
- 破壊的変更が文書化されていない
- 軽微な問題がある
- 改善提案がある

## 重要ルール

- 改善提案（💡）が1件でもある場合、`✅ Approve` や `Approve 相当`、`マージ可能` などの肯定判定を出してはいけない
- 判定・コメント・チェックリストの整合を保ち、改善提案がある場合は必ず `🔄 Request Changes` か `💬 Comment` のいずれかにする
