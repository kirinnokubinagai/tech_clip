# Issue作成ワークフロー

GitHub Issue を標準テンプレートに従って作成する。

## タイトル形式

```
<type>: <説明>
```

| type | 用途 |
|------|------|
| `feat` | 新機能追加 |
| `fix` | バグ修正 |
| `refactor` | リファクタリング |
| `chore` | ビルド・設定・ツール変更 |
| `docs` | ドキュメントのみ |

例:
- `feat: 記事一覧画面にお気に入りフィルターを追加`
- `fix: ログイン後にトークンが保存されないバグを修正`
- `refactor: articlesサービスをリポジトリパターンに分離`

## 本文テンプレート

```markdown
## 概要

（1〜3行で何をするか・なぜ必要かを説明）

## やること

- [ ] 具体的なタスク1
- [ ] 具体的なタスク2
- [ ] テスト追加

## 前提

- 先行Issue: #<番号>（あれば）
- 依存するリソース・外部サービス（あれば）
- ROADMAP の該当Phase（あれば）
```

## 依存関係・関連Issueの記載ルール

- **ブロックされている場合**: `前提: #<番号> が完了していること` を前提セクションに記載
- **関連するだけ**: `関連: #<番号>` を本文末尾に記載
- **ROADMAPとの対応**: `docs/ROADMAP.md` の該当Phaseを確認して記載

## 手順

1. `docs/ROADMAP.md` を確認し、実装順序に違反しないか確認する
2. タイトル・本文をテンプレートに従って作成する
3. 以下の形式で作成する（改行・特殊文字を確実に渡すため heredoc を使う）:
   ```bash
   gh issue create --title "<title>" --body "$(cat <<'EOF'
   ## 概要
   ...
   EOF
   )"
   ```
4. 作成されたIssue番号をユーザーに報告する

## ルール

- Issueなしで実装を始めてはいけない
- タイトルは日本語で具体的に（「〜を改善」「〜を追加」など）
- やることリストは実装者がそのままチェックできる粒度で書く
- 依存関係は必ず明記する（後からわかりにくくなる）
