# エージェント判断ログ（違反事例記録）

このファイルはエージェントが標準ワークフローから外れた独断判断を行った事例を記録する。
今後のエージェント定義改善・プロンプト強化の参考にする。

---

## 2026-04-15 の違反事例

### 違反 1: reviewer が「already resolved」と誤判断して worktree 削除 + PR close

**Issue**: #1013  
**エージェント**: reviewer  
**発生状況**: reviewer が実装内容を確認せずに「すでに解決済み」と判断し、worktree を削除して PR を close した。  
**実際**: 修正が必要な状態だった。  
**影響**: 作業の巻き戻しが必要になり、worktree の再作成と実装の再実行が発生した。  
**根本原因**: reviewer が「既存の変更を確認した」という不完全な情報を「解決済み」と解釈し、AskUserQuestion で確認せずに終了処理を実行した。  
**教訓**: 「resolved」「already fixed」と判定する前に必ず AskUserQuestion で人間に確認する。

---

### 違反 2: orchestrator が conflict を直接 resolve

**Issue**: #983  
**エージェント**: orchestrator  
**発生状況**: origin/main とのコンフリクトを orchestrator が自力で解消し、coder に委譲しなかった。  
**実際**: CLAUDE.md のルールでは conflict 解消は coder に委譲することが定められている。  
**影響**: コンフリクト解消の意図が不明確なコミットが残り、後続レビューで問題が発見された。  
**根本原因**: orchestrator が「修正が簡単だから自分でやる」と判断し、標準委譲フローをスキップした。  
**教訓**: orchestrator は worktree 内のファイルを直接編集しない。コンフリクト解消は必ず coder に委譲する。

---

### 違反 3: reviewer が「改善提案は軽微だから後追いでいい」と誤判断

**Issue**: #996  
**エージェント**: reviewer  
**発生状況**: reviewer が LOW 評価の改善提案を「軽微だから後追いで対応すればいい」と判断し、PASS を出した。  
**実際**: ユーザーは LOW 評価であっても全件修正を期待していた。  
**影響**: ユーザーが激怒し、追加修正対応が必要になった。  
**根本原因**: reviewer が「重要度が低い指摘は省略可能」と自己解釈し、CLAUDE.md の「CRITICAL / HIGH / MEDIUM / LOW すべての指摘が 0 件になるまで PASS を出さない」ルールを無視した。  
**教訓**: LOW 評価であっても指摘が 1 件でも残れば PASS を出さない。「軽微だから省略する」の自己判断は禁止。

---

### 違反 4: coder が別 PR のコミットと自ブランチのコミットを誤識別

**Issue**: #994  
**エージェント**: coder (v2)  
**発生状況**: coder がコミットハッシュ d4ee36cc を「別 PR のコミット」と誤識別し、自ブランチのコミットであるにもかかわらず別扱いした。  
**実際**: d4ee36cc は #994 ブランチのコミットだった。  
**影響**: 実装の重複やコミット管理の混乱が発生した。  
**根本原因**: coder が git log の出力を正確に確認せず、「文脈的に別 PR のコミットだろう」と決めつけた。  
**教訓**: コミットの帰属が不明確な場合は git log --oneline で確認し、それでも判断できない場合は AskUserQuestion で確認する。

## 追記の運用ルール

違反事例の追記は以下のタイミングで行う:

- reviewer / infra-reviewer / ui-reviewer が CHANGES_REQUESTED を出したタイミング
- orchestrator が AskUserQuestion を呼んで人間判断を仰いだタイミング
- 標準ワークフロー逸脱が事後に発覚したタイミング

追記は発見した担当エージェントが行う。追記後は orchestrator に SendMessage で通知する。

---

## 記録フォーマット

新しい違反事例が発生した場合、以下の形式で追記すること:

```markdown
### 違反 N: <違反内容の一行サマリー>

**Issue**: #<番号>
**エージェント**: <エージェント名>
**発生状況**: <何が起きたか>
**実際**: <本来はどうあるべきだったか>
**影響**: <何が起きたか、どんな被害が発生したか>
**根本原因**: <なぜこの判断に至ったか>
**教訓**: <今後どうすれば防げるか>
```
